## 2026-03-10 工作记录
### 问题1：包路径错误
**问题描述**：原项目使用的包路径是`jijizhazha1024/go-mall`，但现在仓库已迁移到`https://github.com/falconfan123/Go-mall.git`，需要更新所有包路径。
**解决过程**：
1. 修改`go.mod`文件中的模块路径为`module github.com/falconfan123/Go-mall`
2. 批量替换所有Go文件中的`jijizhazha1024/go-mall`为`github.com/falconfan123/Go-mall`
3. 在`go.mod`中添加replace语句，将内部服务的导入路径指向本地目录
### 问题2：protobuf生成目录名不规范
**问题描述**：部分服务的protobuf生成目录名与导入路径不一致，导致编译错误：
- product服务的pb目录名为`productcatalogservice`，应该为`product`
- order服务的pb目录名为`orderservice`，应该为`order`
**解决过程**：
1. 重命名`productcatalogservice`目录为`product`
2. 重命名`orderservice`目录为`order`
3. 批量替换所有引用中的`productcatalogservice`为`product`
4. 批量替换所有引用中的`orderservice`为`order`
### 问题3：循环导入问题
**问题描述**：users服务出现循环导入错误，原因是服务启动文件导入了和目录同名的包。
**解决过程**：
1. 检查目录结构，发现pb文件的组织方式存在问题，需要调整目录结构
2. 将 `users` 服务下的 `users` 目录（存放pb生成代码）重命名为 `userspb`，并修改 `users.proto` 中的 `go_package` 为 `./userspb`。
3. 更新 `users.go` 和其他引用文件中的导入路径，指向新的 `userspb` 包。
4. 重新生成代码：`goctl rpc protoc users.proto --go_out=. --go-grpc_out=. --zrpc_out=.`。
5. 清理旧的 `internal/logic` 和 `internal/server` 代码，确保与新生成的代码一致。

### 问题4：Users 服务逻辑缺失与数据库不匹配
**问题描述**：
1. `users` 服务的 `Register`, `Login`, `Logout` 等核心逻辑为空（生成的模板代码）。
2. 数据库表 `users` 缺少 `logout_at` 和 `login_at` 字段，导致代码运行时 SQL 报错。
3. 缺少部分错误码定义（如 `UserNotExistError`, `LoginError`）。

**解决过程**：
1. **数据库迁移**：在 MySQL `users` 表中添加 `logout_at` 和 `login_at` 字段。
2. **常量补充**：在 `common/consts/code/user.go` 中补充缺失的错误码和错误信息。
3. **逻辑实现**：
    - **Register**：实现了邮箱查重、密码加密、用户创建、双 Token (Access/Refresh) 生成。
    - **Login**：实现了用户存在性检查、密码比对、Token 生成。
    - **Logout**：实现了更新数据库中的登出时间。
    - **UpdateUser**：实现了用户名和头像的更新。
    - **DeleteUser**：实现了用户的软删除。
    - **GetUser**：实现了用户信息的查询。
4. **验证**：使用 `grpcurl` 对上述接口进行了测试，均返回成功或预期的错误码。

### 问题5：Product 服务库存与价格问题
**问题描述**：
1. `products` 表结构与 Go 结构体不一致：数据库 `products` 表缺少 `stock` 字段（或模型未包含），且 `price` 字段类型定义不明确（DB为Decimal，Proto为Int64）。
2. `CreateProduct` 接口调用成功但未初始化库存。
3. `GetProduct` 接口返回的 `stock` 字段为 0，原因是 `Inventory` 服务虽然连接成功但未同步库存数据，且 `GetProductLogic` 逻辑覆盖了 DB 中的库存数据。
4. `GetProduct` 接口缺少 `ProductNotExistError` 错误码。

**解决过程**：
1. **数据库调整**：
    - 将 `products` 表的 `price` 字段修改为 `BIGINT`（单位：分），并更新现有数据（x100）。
    - 确认 `products` 表包含 `stock` 字段。
    - 更新 `dal/model/products/product/product.sql` 并重新生成 `productsmodel_gen.go` 模型代码。
2. **逻辑完善**：
    - 修改 `CreateProductLogic` 和 `UpdateProductLogic`，在操作 DB 后同步调用 `InventoryRpc.UpdateInventory` 以初始化或更新库存。
    - 修改 `GetProductLogic`：
        - 增加对 `InventoryRpc` 调用的超时控制（500ms），防止阻塞。
        - 优化库存获取逻辑：优先使用 DB 中的库存，仅当 `InventoryRpc` 调用成功时才覆盖库存值。
        - 补充 `ProductNotFound` 错误码处理。
3. **验证**：
    - `CreateProduct`：成功创建商品并同步库存。
    - `GetProduct`：成功获取商品详情，包括正确的 `stock` 值。
    - `UpdateProduct`：成功更新商品信息并清理缓存。

### 问题6：Checkout 服务配置与逻辑缺陷
**问题描述**：
1. **服务发现失败**：`checkout` 无法连接 `product` 服务，报错 `produced zero addresses`。原因是 `product` 服务注册名为 `products.rpc`（复数），而 `checkout.yaml` 配置为 `product.rpc`（单数）。
2. **数据库结构缺失**：`checkouts` 表缺少 `pre_order_id`、`address_id` 等字段；`checkout_items` 表缺少 `pre_order_id` 且主键定义错误（不能以 `pre_order_id` 为主键，因多商品会冲突）。
3. **资源未释放**：`PrepareCheckout` 接口在 `ProductRpc` 或其他下游服务调用失败时，未回滚已扣减的库存（Inventory），也未释放 Redis 锁，导致后续重试时 Inventory 返回 "订单已处理"。
4. **编译错误**：`checkout` 引用的 `product` 客户端代码包路径不匹配（`goctl` 生成目录变更导致）。

**解决过程**：
1. **配置修正**：修改 `checkout.yaml`，将 `ProductRpc` 目标修正为 `consul://localhost:8500/products.rpc`。
2. **数据库重构**：
    - 修改 `checkout.sql`，完善 `checkouts` 表字段。
    - 重构 `checkout_items` 表，添加自增主键 `id`，将 `pre_order_id` 改为普通索引。
    - 重建数据库表并重新生成 Model 代码。
3. **逻辑完善**：
    - 在 `PrepareCheckoutLogic` 中添加 `rollback` 方法，用于在下游失败时同步回滚库存并释放 Redis 锁。
    - 修复 Go 代码中 Model 调用参数（适配无缓存 Model 构造函数）。
4. **验证**：`PrepareCheckout` 接口测试通过，成功生成预订单 ID，并正确扣减库存。

### 问题7：Order 服务与 Users 服务集成问题
**问题描述**：
1. **Client 代码缺失**：`order` 服务无法引用 `usersclient` 包，因为 `users` 服务重构后 Client 代码位置变更。
2. **Users 服务逻辑缺失**：`users` 服务未实现 `GetAddress` 和 `AddAddress` 接口，导致 `order` 服务调用时 panic 或返回空。
3. **数据库 Model 冲突**：`users` 服务重新生成代码后，`ServiceContext` 丢失了 `UsersModel`，且 `UserAddressesModel` 构造函数签名不匹配（缓存配置问题）。
4. **代码重复**：`users` 服务中存在大量重复的 Logic 和 Server 文件（`xxx_logic.go` vs `xxxlogic.go`），导致编译错误。

**解决过程**：
1. **Client 代码修复**：重新生成 `users` 服务的 Client 代码，并更新 `order` 服务的引用为 `users` 包。
2. **逻辑实现**：
    - 在 `users` 服务中实现了 `AddAddress` 和 `GetAddress` 逻辑。
    - 修复了 `getaddresslogic.go` 中的类型转换错误（`int64` vs `uint64`）。
3. **环境修复**：
    - 清理 `users` 服务中重复的 Go 文件。
    - 修正 `ServiceContext` 初始化代码，正确传入 `UserAddressesModel`。
4. **验证**：
    - 成功调用 `AddAddress` 添加收货地址。
    - `CreateOrder` 接口成功调用 `users.GetAddress` 和 `checkout.GetCheckoutDetail`。
    - 最终 `CreateOrder` 返回 "订单已存在"（符合幂等性预期），证明全链路打通。
