# 前端实现要求文档（PRD_frontend.md）
## 一、功能概述
本方案实现基于长短令牌的用户登录态管理机制，确保用户登录态安全、自动续期，同时支持服务端强制用户下线功能。

## 二、核心流程实现要求

### 1. 登录流程
- 用户提交登录请求（账号密码/短信/第三方登录等）
- 登录成功后，服务端返回 `long_token` 和 `short_token`
- 前端需要将两个令牌持久化存储：
  - 优先存储在 `HttpOnly` + `Secure` Cookie 中（防止XSS攻击）
  - 如果使用LocalStorage存储，需要添加XSS防护措施
- 存储时需要记录 `short_token` 的过期时间（从令牌中解析或单独存储）

### 2. 请求头携带规则
- **所有接口请求**都必须在HTTP请求头中携带两个令牌：
  ```http
  long_token: [长令牌值]
  short_token: [短令牌值]
  ```
- 注意：登录、注册接口不需要携带令牌

### 3. 短令牌自动更新机制
- 每次请求响应时，检查响应头中是否存在 `Short-Token-Refresh` 字段
- 如果存在该字段，说明短令牌已过期，服务端返回了新的短令牌
- 前端需要自动更新本地存储的 `short_token`，替换原有值
- 整个更新过程对用户完全透明，无感知

### 4. 错误处理规则
- 当接口返回 `401 Unauthorized` 状态码时：
  - 如果错误信息为 `令牌非法` 或 `Session不存在`：清除本地存储的令牌，跳转到登录页面
  - 如果错误信息为 `短令牌过期`：自动重试当前请求（使用新的短令牌）
- 当接口返回 `403 Forbidden` 状态码时：说明用户无权限访问该资源，提示用户即可，不需要跳转登录

### 5. 登出流程
- 用户主动登出时，调用登出接口
- 登出成功后，清除本地存储的所有令牌信息
- 跳转到登录页面

### 6. 设备ID生成规则
- 首次打开应用时，生成唯一的设备ID（UUID）
- 设备ID存储在本地，永久有效（除非用户卸载应用/清除浏览器数据）
- 登录请求时需要在请求参数中携带 `device_id` 字段
- 设备ID用于短令牌生成和多设备登录管控

## 三、安全要求
1. 禁止将令牌明文输出到控制台日志
2. 禁止在URL参数中传递令牌
3. 跨域请求时需要配置 `withCredentials: true` 携带Cookie
4. 前端需要开启CSP防护，防止XSS攻击窃取令牌
5. 短令牌过期时间建议在前端做倒计时提示，提前10分钟主动调用刷新接口

## 四、测试用例
| 测试场景 | 预期结果 |
|---------|---------|
| 正常登录 | 成功获取长短令牌，存储到本地 |
| 携带令牌请求接口 | 接口正常返回数据 |
| 短令牌过期后请求接口 | 服务端自动返回新的短令牌，前端自动更新，请求成功 |
| 长令牌过期后请求接口 | 返回401，跳转到登录页面 |
| 主动登出 | 本地令牌被清除，跳转到登录页面 |
| 服务端强制下线 | 用户下一次请求时返回401，跳转到登录页面 |
| 多设备登录 | 不同设备的登录态互不影响，单独管控 |
